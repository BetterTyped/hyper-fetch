The biggest difference of **`Hyper Fetch`** compared to other libraries is the architecture and assumptions on which it
is built. It is a tool whose core features are written in js without dependencies, thanks to which it can work in many
environments without a specific connection to a given framework, offering full control over the flow and observation of
data exchange, it is not just a wrapper. The main goal was to introduce a data storage standard, offer an embedded http
client, reduce setup time and solve major architectural difficulties.

:::info

Producing an accurate and unbiased comparison is quite a challenge. Libraries develop very dynamically and quickly, so
if you notice that our data may be corrected while using these libraries please let us know by opening an issue.

:::

## Why it's Unique

- Easy Upload / Download progress / ETA tracking
- Requests queueing and many [dispatching strategies](/docs/Architecture/Dispatcher#dispatching-modes)!
- [Command](/docs/Architecture/Command) / [Builder](/docs/Architecture/Builder) pattern gives you amazing control over
  requests at any given time
- Architecture prepared for code sharing which gives possibilities for
  [testers to hook into development setup and types](/docs/Getting%20Started/Testing)
- Flat side-effects [helper hooks annotation](/docs/React/Overview#helper-hooks) and focus on readable code
- [Recipes for structure](/docs/Getting%20Started/Development#structure) for large scale applications which dramatically
  limits the maintainance costs
- Well tested and prepared from years of research and experimenting
- [Offline](/guides/Advanced/Offline) / [Persistence](/guides/Advanced/Persistence) as core features, which gives you
  full control over your data!
